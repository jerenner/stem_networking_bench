# Current STEM Packet Format

This is the complete sender-facing format for the FPGA test stream currently
accepted by the DOEELS receivers.

## Current Packet

Each detector packet is an untagged Ethernet II/IPv4/UDP jumbo frame containing
a 64-byte custom STEM header followed by 3,840 detector samples.

```text
Complete Ethernet frame as delivered to DPDK, excluding the Ethernet FCS

byte 0                                                            byte 7785
  |                                                                   |
  +----------+--------+------+----------------+------------------------+
  | Ethernet | IPv4   | UDP  | STEM header    | detector samples       |
  | 14 B     | 20 B   | 8 B  | 64 B           | 7,680 B                |
  +----------+--------+------+----------------+------------------------+
  0         14       34     42              106                     7786
```

| Packet bytes | Size | Contents |
|---|---:|---|
| `0..13` | 14 B | Standard Ethernet II header |
| `14..33` | 20 B | Standard IPv4 header, with no IPv4 options |
| `34..41` | 8 B | Standard UDP header |
| `42..105` | 64 B | Custom STEM header described below |
| `106..7785` | 7,680 B | 3,840 little-endian unsigned 16-bit samples |

The corresponding protocol lengths are:

| Quantity | Bytes |
|---|---:|
| UDP data: STEM header plus detector samples | 7,744 |
| UDP length field: UDP header plus UDP data | 7,752 |
| IPv4 total-length field | 7,772 |
| Ethernet frame delivered to DPDK, without FCS | 7,786 |
| Ethernet frame on Ethernet, including 4-byte FCS | 7,790 |

The Ethernet FCS (Frame Check Sequence) is a standard 4-byte CRC-32 appended
by the transmitting Ethernet MAC. The receiving NIC verifies it and normally
removes it before giving the packet to DPDK, so it is not included in the
7,786-byte application-visible packet and does not need to be generated by the
STEM application code. Ethernet preamble/SFD and the inter-packet gap are also
handled below the application packet interface.

The 42-byte outer-header layout is confirmed by three independent properties
of the running system: the RX hardware flow matches UDP destination port
`23130`, the receiver successfully parses the STEM fields beginning at byte
42, and the NIC statistics report approximately 7,786 received bytes per
packet. Packets without valid Ethernet, IPv4, and UDP headers would not match
the configured UDP flow.

The receiver selects packets by UDP destination port `23130`. MAC addresses,
IP addresses, and UDP source port are deployment-specific rather than part of
the STEM data format. The network must not add a VLAN tag or IPv4 options,
because the receiver assumes exactly 42 bytes before the STEM header. IPv4
fragmentation is not supported.

Ethernet, IPv4, and UDP fields use their standard network-byte-order encoding.
Detector samples use little-endian `uint16` encoding. Ethernet FCS generation,
IPv4 checksum generation, and any enabled UDP checksum generation are normal
Ethernet/IP/UDP responsibilities rather than custom STEM fields.

## Custom 64-Byte STEM Header

The custom header starts at Ethernet-frame byte 42. Custom-header offsets below
are relative to that position; the table also gives absolute packet offsets.

```text
STEM-header offset

  0      2      4      6      8     10                              64
  +------+------+------+------+------+--------------------------------+
  |5A5A  |frame |row   |eth ID|A5A5  | 27 x 16-bit loop-counter words |
  |2 B   |2 B   |2 B   |2 B   |2 B   | 54 B                           |
  +------+------+------+------+------+--------------------------------+
```

| STEM offsets | Packet offsets | Size | Encoding | Value |
|---|---:|---:|---|---|
| `0..1` | `42..43` | 2 B | 16-bit marker | `0x5A5A` |
| `2..3` | `44..45` | 2 B | Unsigned 16-bit | FPGA `frame_count` |
| `4..5` | `46..47` | 2 B | Unsigned little-endian | FPGA `row_count`; read by software as `row_number` |
| `6..7` | `48..49` | 2 B | Unsigned little-endian | FPGA `eth_id`; read by software as `source_id` |
| `8..9` | `50..51` | 2 B | 16-bit marker | `0xA5A5` |
| `10..63` | `52..105` | 54 B | 27 16-bit words | FPGA loop-counter data |

All 64 header bytes are therefore populated by the FPGA; the previously
reserved regions must not be assumed to contain zero. The current receiver
uses only `row_count` and `eth_id` for frame assembly. It ignores both marker
words, the separate `frame_count`, and the loop-counter words.

To generate the current stream, use:

```text
row_number = (frame_index modulo 128) * 128 + packet_position

frame_index      advances once per detector frame
packet_position  runs from 0 through 127 independently for each source_id
row_number       therefore runs from 0 through 16383 and then wraps to 0
```

Up to eight source IDs, `0..7`, can contribute to one frame on one physical
receiver interface. A full eight-source
stream carries `8 * 128 = 1,024` packets per frame interval.

Packets may arrive out of order because the receiver reconstructs frame and
packet identity from `row_number` and `source_id`. At startup, at least one
enabled source must eventually send `row_number = 0` so the receiver can
synchronize to the 128-frame counter cycle.
